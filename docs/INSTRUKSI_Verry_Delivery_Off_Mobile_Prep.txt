BAGIAN 3 — SIAPKAN MOBILE APP (sambil menunggu Stripe, 2-3 minggu)  [AKTIF]
================================================================================================
Manfaatkan waktu tunggu untuk menyiapkan mobile app. Prinsip WAJIB: mobile app harus mengikuti
arsitektur MULTI-TENANT yang sama (satu app, banyak tenant / atau per-tenant dari config),
BUKAN app baru per restoran yang di-hardcode.

Keputusan model: Opsi B (final) — lihat docs/INSTRUKSI_Verry_Mobile_App_OpsiB.txt.

Yang perlu diklarifikasi & direncanakan (diskusikan dgn Malik):
1. MODEL APP:
   - Opsi A: SATU app multi-tenant (customer pilih/deteksi restoran) — mirih platform.
   - Opsi B: App per-restoran yang di-generate dari config (tiap restoran punya app sendiri
     di store, tapi dari SATU codebase + config tenant) — mirip "white-label app" Owner.com.
   Owner.com/DoorDash pola: umumnya per-restoran branded app dari satu platform. Malik perlu
   putuskan A atau B. (Rekomendasi: samakan dengan strategi web — config-driven, satu codebase.)
   → DIPUTUSKAN: Opsi B (lihat docs/INSTRUKSI_Verry_Mobile_App_OpsiB.txt).

2. FUNGSI MVP MOBILE (samakan dengan web, PICKUP dulu):
   - Lihat menu (dari Square catalog tenant), tambah ke cart, checkout PICKUP, bayar CARD.
   - Order status / konfirmasi. Login/identifikasi customer sederhana.
   - Delivery DITUNDA (sama seperti web) sampai Stripe siap.
   - Anchor BP mengikuti mode tenant (pos-native/platform) — sama seperti web.

3. ARSITEKTUR:
   - Pakai BACKEND yang SAMA dengan web (endpoint yang sama), jangan bikin backend terpisah.
   - Kredensial di backend/secrets, TIDAK di app (app tidak boleh pegang API key POS/Stripe/BP).
   - Config-driven: identitas tenant (logo, warna, menu) dari config, seperti web.
   - Teknologi: sarankan (mis. React Native/Expo) yang bisa berbagi logika dengan web bila memungkinkan.

4. JANGAN sambungkan pembayaran Stripe di app dulu — Stripe belum live. Pickup + Square dulu
   (atau siapkan kerangka pembayaran yang mudah diganti ke Stripe nanti).
